# Phase 1.6c — 多 Host 管理

SPA 支援同時連線多台 daemon，提供 Host 管理頁面（連線、config、sessions、hooks、上傳暫存），並將 #128 session 即時同步納入多 host 架構。

## 範圍

1. **Host Store + 多 daemon 連線** — SPA 管理 host 清單，每個 host 獨立連線
2. **Session Store 改 nested** — `Record<hostId, Session[]>`，PaneContent 加 hostId
3. **API/WS 多 host 化** — 請求依 hostId 路由到對應 daemon，每個 host 各一條 Event WS
4. **Host 頁面** — B+C 混合佈局：左側 accordion + 總覽長頁 + 獨立子頁
5. **API Token 認證** — daemon auth middleware + SPA onboarding 流程
6. **#128 Session 即時同步** — tmux hooks + WS 推送，多 host 架構下設計
7. **Session 管理** — Host 頁新建/刪除 session，new-tab 清單按 host 分組
8. **離線處理** — 快取最後狀態 + disable + reconnecting 回饋 + tab 斷線 icon

### 不做（延後）

- CORS 白名單管理（#112 部分，獨立處理）
- Session 共享連結 / 唯讀掛載模式（1.8 後段）
- Tab split + Pane attach（1.9）

## §1: Host Store

### 資料結構

```typescript
interface HostConfig {
  id: string          // generateId()
  name: string        // 使用者自訂名稱
  ip: string          // e.g. "100.64.0.2"
  port: number        // e.g. 7860, 預設 7860
  token?: string      // API token（daemon 未啟用時為 undefined）
  order: number       // 排序
}

// 執行時狀態（不 persist）
interface HostRuntime {
  status: 'connected' | 'disconnected' | 'reconnecting'
  latency?: number    // ms, 來自 health check
  info?: HostInfo     // daemon 回報的系統資訊
}

interface HostInfo {
  tbox_version: string
  tmux_version: string
  os: string
  arch: string
}
```

### useHostStore

Zustand + persist。

```typescript
interface HostState {
  hosts: Record<string, HostConfig>       // persisted
  hostOrder: string[]                     // persisted, 排序用
  runtime: Record<string, HostRuntime>    // 不 persist
  activeHostId: string | null             // persisted, Host 頁面用

  addHost: (config: Omit<HostConfig, 'id' | 'order'>) => string
  updateHost: (id: string, updates: Partial<HostConfig>) => void
  removeHost: (id: string) => void
  reorderHosts: (ids: string[]) => void
  setActiveHost: (id: string | null) => void
  setRuntime: (id: string, runtime: Partial<HostRuntime>) => void
}
```

persist name: `tbox-hosts`，partialize 排除 `runtime`。

### Helper

```typescript
function getHostBase(host: HostConfig): string {
  return `http://${host.ip}:${host.port}`
}

function getHostWsBase(host: HostConfig): string {
  return `ws://${host.ip}:${host.port}`
}
```

### 自動遷移

App 啟動時，如果 `hostOrder.length === 0`，自動建立一筆 host：

- name: `"Local"`
- ip: 從現有 `VITE_DAEMON_BASE` env var 解析，fallback `"100.64.0.2"`
- port: 從 env var 解析，fallback `7860`

現有 session store 的 sessions 遷移到 `sessions[autoHostId]`。PaneContent 裡的 session kind 加上 `hostId: autoHostId`。

## §2: Session Store 改 Nested

### 結構變更

```typescript
// Before
interface SessionState {
  sessions: Session[]
  activeId: string | null
  fetch: (base: string) => Promise<void>
  setActive: (code: string | null) => void
}

// After
interface SessionState {
  sessions: Record<string, Session[]>    // hostId → sessions
  activeHostId: string | null            // 目前 active session 的 host
  activeCode: string | null              // 目前 active session 的 code
  fetchHost: (hostId: string, base: string) => Promise<void>
  replaceHost: (hostId: string, sessions: Session[]) => void
  removeHost: (hostId: string) => void
  setActive: (hostId: string | null, code: string | null) => void
}
```

persist name 維持 `tbox-sessions`，加 version migration。

### 存取模式改法

| 原本 | 改為 |
|------|------|
| `sessions.find(s => s.code === code)` | `sessions[hostId]?.find(s => s.code === code)` |
| `sessions.map(...)` | `(sessions[hostId] ?? []).map(...)` |
| `sessions.length === 0` | `(sessions[hostId] ?? []).length === 0` |
| `set({ sessions })` | `replaceHost(hostId, data)` |

所有使用 session store 的元件（6+ find、3 map、1 filter、2 length）需一併修改，hostId 來源統一從 PaneContent 或 Host context 取得。

### PaneContent 變更

```typescript
// Before
| { kind: 'session'; sessionCode: string; mode: 'terminal' | 'stream' }

// After
| { kind: 'session'; hostId: string; sessionCode: string; mode: 'terminal' | 'stream' }
```

新增：

```typescript
| { kind: 'hosts' }   // Host 管理頁面 singleton
```

## §3: API/WS 多 Host 化

### API 層

`api.ts` 已經每個 function 都吃 `base: string`，不需改簽名。呼叫端從 host store 查 base URL：

```typescript
const host = useHostStore.getState().hosts[hostId]
const base = getHostBase(host)
const sessions = await listSessions(base)
```

Token 注入：所有 `fetch` 呼叫統一透過 wrapper：

```typescript
function hostFetch(hostId: string, path: string, init?: RequestInit): Promise<Response> {
  const host = useHostStore.getState().hosts[hostId]
  const base = getHostBase(host)
  const headers = new Headers(init?.headers)
  if (host.token) {
    headers.set('Authorization', `Bearer ${host.token}`)
  }
  return fetch(`${base}${path}`, { ...init, headers })
}
```

### WS 層

每個 host 各一條 Event WS（`/ws/session-events`）。WS 連線也需攜帶 token（透過 URL query param `?token=...`，因 WebSocket API 不支援自訂 header）。

```typescript
function hostWsUrl(hostId: string, path: string): string {
  const host = useHostStore.getState().hosts[hostId]
  const base = getHostWsBase(host)
  const url = new URL(`${base}${path}`)
  if (host.token) url.searchParams.set('token', host.token)
  return url.toString()
}
```

### useSessionEventWs 改為多 host

現在接收 `wsBase` + `daemonBase`。改為：

```typescript
function useMultiHostEventWs() {
  const hosts = useHostStore((s) => s.hosts)
  const hostOrder = useHostStore((s) => s.hostOrder)

  useEffect(() => {
    const connections = new Map<string, ReturnType<typeof connectSessionEvents>>()

    for (const hostId of hostOrder) {
      const host = hosts[hostId]
      if (!host) continue
      const wsUrl = hostWsUrl(hostId, '/ws/session-events')
      const conn = connectSessionEvents(wsUrl, (event) => {
        // 同現有邏輯，但 session lookup 帶 hostId
        handleEvent(hostId, event)
      }, /* onClose */ () => {
        useHostStore.getState().setRuntime(hostId, { status: 'disconnected' })
      }, /* onOpen */ () => {
        useHostStore.getState().setRuntime(hostId, { status: 'connected' })
        useAgentStore.getState().clearSubagentsForHost(hostId)
      })
      connections.set(hostId, conn)
    }

    return () => { connections.forEach((c) => c.close()) }
  }, [hosts, hostOrder])
}
```

## §4: Daemon Auth Middleware

### Token 驗證

```go
func authMiddleware(token string, next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if token == "" {
            next.ServeHTTP(w, r)   // 未設定 token 時不驗證
            return
        }
        // HTTP: Authorization header
        // WS: ?token= query param
        var provided string
        if auth := r.Header.Get("Authorization"); strings.HasPrefix(auth, "Bearer ") {
            provided = strings.TrimPrefix(auth, "Bearer ")
        } else {
            provided = r.URL.Query().Get("token")
        }
        if subtle.ConstantTimeCompare([]byte(provided), []byte(token)) != 1 {
            http.Error(w, "unauthorized", http.StatusUnauthorized)
            return
        }
        next.ServeHTTP(w, r)
    })
}
```

### 例外路徑

以下 endpoint 不需驗證（允許無 token 存取）：

- `GET /api/health` — 連線測試
- `POST /api/token/generate` — 產生 token（僅限本機 IP）

### `tbox token generate` CLI

```
$ tbox token generate

```

## §6: #128 Session 即時同步

直接在多 host 架構下設計，每個 host 的 daemon 獨立推送。

### Daemon 端

Session module 新增 watcher goroutine（`Start()` 內啟動）：

```go
func (m *Module) watchSessions(ctx context.Context) {
    // goroutine A: tmux hooks 即時推送
    go func() {
        for {
            // tmux wait-for sess_evt（阻塞直到收到信號）
            cmd := exec.CommandContext(ctx, "tmux", "wait-for", "sess_evt")
            if err := cmd.Run(); err != nil {
                if ctx.Err() != nil { return }
                time.Sleep(time.Second)
                continue
            }
            m.broadcastSessions()
        }
    }()
    // goroutine B: polling fallback（5s）
    go func() {
        ticker := time.NewTicker(5 * time.Second)
        defer ticker.Stop()
        var lastHash string
        for {
            select {
            case <-ctx.Done(): return
            case <-ticker.C:
                hash := m.sessionsHash()
                if hash != lastHash {
                    lastHash = hash
                    m.broadcastSessions()
                }
            }
        }
    }()
}
```

### tmux hooks 安裝

daemon 啟動時自動安裝（如果 tmux 可用）：

```bash
tmux set-hook -g session-created  "run-shell 'tmux wait-for -S sess_evt'"
tmux set-hook -g session-closed   "run-shell 'tmux wait-for -S sess_evt'"
tmux set-hook -g session-renamed  "run-shell 'tmux wait-for -S sess_evt'"
```

daemon 停止時移除（`Stop()` 內）。

### WS 事件

Event WS 新增 `sessions` 事件類型：

```json
{
  "type": "sessions",
  "session": "",
  "value": "[{\"code\":\"a1b2c3\",\"name\":\"dev\",...}]"
}
```

`value` 是整個 sessions array 的 JSON 字串（與 `GET /api/sessions` 回傳格式相同）。

### SPA 端

`useMultiHostEventWs` 收到 `sessions` 事件時：

```typescript
if (event.type === 'sessions') {
  const data: Session[] = JSON.parse(event.value)
  useSessionStore.getState().replaceHost(hostId, data)
}
```

移除 App.tsx 的啟動 `fetchSessions()`。改為在 Event WS `onOpen` 時主動 fetch 一次（確保有初始資料）。

## §7: 離線處理

### 偵測機制

Event WS 的 onClose / onError 觸發離線判定。重連邏輯沿用現有 `connectSessionEvents` 的 exponential backoff。

### 各 UI 位置行為

| 位置 | 離線行為 |
|------|----------|
| **StatusBar** | `connected` → `disconnected` / `reconnecting...` |
| **Session list（new-tab）** | disable 該 host 所有 session item，host 名旁顯示 `reconnecting...` |
| **Host 側欄** | 🟢 → 🔴 或轉圈 icon |
| **Host 總覽** | 連線狀態顯示 disconnected / reconnecting... |
| **Host Sessions 子頁** | 清單保留但 disable，「New Session」disable |
| **已開 tab（Terminal/Stream）** | 現有斷線 overlay + reconnect 文字 |
| **SortableTab 標題** | 顯示斷線 icon（Phosphor `WifiSlash` 或 `LinkBreak`） |
| **通知** | 該 host 的 agent hook 暫停，重連後不補發 |

### Session 資料快取

離線時保留 `sessions[hostId]` 最後已知資料，不清空。重連成功後由 WS 事件或初始 fetch 覆蓋。

## §8: Host Onboarding 流程

### 新增 Host

1. 使用者點 `+ Add Host`
2. 填寫 IP + Port + Name（可選）
3. 點「連線測試」
4. SPA 呼叫 `GET /api/health`
   - **200** → 連線成功，嘗試 `GET /api/sessions`
     - **200** → daemon 無需 token，完成
     - **401** → daemon 要求 token，顯示 token 輸入欄位
   - **連線失敗** → 顯示錯誤訊息，可重試
5. 有 token 需求時：輸入 token → 帶 Authorization header 重試 → 驗證通過 → 完成
6. 儲存到 host store

### Token 設定引導

如果 daemon 尚未設定 token：
- 使用者在 daemon 主機執行 `tbox token generate`
- 複製產生的 token 到 SPA 的 host 設定

## §10: Persist Migration

### 初始化順序

Host store 必須最先初始化（Zustand persist hydrate），其他 store 的 migration 才能讀到 `hostOrder[0]`。

啟動順序：
1. Host store hydrate + 自動遷移（如果 hostOrder 為空，建立 Local host）
2. Session store hydrate + migration（讀 hostOrder[0]）
3. Tab store hydrate + migration（讀 hostOrder[0]）
4. 其他 store

實作方式：App.tsx 加 hydration gate，等 host store ready 後再 render 其他元件（沿用現有 Zustand persist hydration 等待模式）。

### Session Store（version 1 → 2）

```typescript
migrate: (persisted: any, version: number) => {
  if (version < 2) {
    // 單 host → 多 host
    const defaultHostId = useHostStore.getState().hostOrder[0]
    return {
      ...persisted,
      sessions: { [defaultHostId]: persisted.sessions ?? [] },
      activeHostId: defaultHostId,
      activeCode: persisted.activeId ?? null,
    }
  }
  return persisted
}
```

### Tab Store（PaneContent migration）

掃描所有 tab 的 PaneContent，`kind === 'session'` 的加上 `hostId`：

```typescript
migrate: (persisted: any, version: number) => {
  if (version < N) {
    const defaultHostId = useHostStore.getState().hostOrder[0]
    // walk all tabs → layout → pane → content
    // if kind === 'session' && !hostId → add hostId = defaultHostId
  }
  return persisted
}
```

### Upload Store

`useUploadStore` 的 key 從 `sessionCode` 改為 `${hostId}:${sessionCode}`（runtime only，不 persist，不需 migration）。

## §11: Stream Store 多 Host 化

Stream store 的 per-session state（messages、relayStatus、handoffProgress 等）key 也需加 hostId。

改為 `${hostId}:${sessionCode}` 作為 composite key（runtime only，不 persist）。

所有存取 stream store 的地方傳入 hostId。

## §12: Agent Store 多 Host 化

Agent store 的 key 目前是 sessionCode。多 host 後需避免碰撞。

改為 `${hostId}:${sessionCode}` 作為 composite key（runtime only，不 persist events/statuses）。

所有存取 agent store 的地方傳入 hostId，store 方法簽名加 hostId 參數：

```typescript
handleHookEvent: (hostId: string, sessionCode: string, hookData: any) => void
getStatus: (hostId: string, sessionCode: string) => AgentStatus | undefined
```
